# PRD: 사내 AI PM 어시스턴트 (Paimy) v2.0

## 1. 제품 개요 (Overview)

### 1.1 제품명

- **사내 AI PM 어시스턴트 – Paimy**

### 1.2 문제 정의 (Problem Statement)

현재 사내 프로젝트 및 액션 아이템 관리는 노션을 중심으로 이루어지고 있으나,

- 슬랙 ↔ 노션 간 **컨텍스트 전환 비용**이 높고
- 업무 상태 업데이트가 **지연 또는 누락**되며
- PM 및 팀원들이 **현재 우선순위와 진행 상황을 실시간으로 파악하기 어려움**
- **이메일에서 발생한 액션 아이템**이 체계적으로 추적되지 않음
- **캘린더 일정과 태스크 마감일** 간의 연계가 부족하여 일정 충돌 및 누락 발생

이로 인해 프로젝트 추진력이 저하되고, PM의 운영 부담이 증가하고 있음.

### 1.3 제품 비전 (Vision)

> "슬랙 대화만으로 프로젝트가 굴러가게 한다."

Paimy는 슬랙 대화를 인터페이스로 삼아
**노션 + Google Workspace(캘린더, Gmail)** 기반 프로젝트 관리의 **마찰을 제거하고 실행력을 극대화하는 AI PM 어시스턴트**다.

## 2. 목표 및 성공 지표 (Goals & Metrics)

### 2.1 제품 목표

- 프로젝트 관리에서 발생하는 **운영 오버헤드 최소화**
- 액션 아이템의 **상태 최신성(Freshness) 확보**
- PM 및 팀원의 **실행 중심 커뮤니케이션 강화**
- **이메일 기반 액션 아이템의 체계적 추적**
- **일정-태스크 간 통합 관리**를 통한 업무 가시성 확보

### 2.2 성공 지표 (예시)

| 지표 | 설명 |
|------|------|
| 슬랙을 통한 태스크 상태 변경 비율 | 전체 상태 변경 중 Paimy 경유 비율 |
| 마감 기한 경과 태스크 비율 감소 | 리마인드 기능 효과 측정 |
| 주간 리포트 수동 작성 시간 감소 | 자동 브리핑 효과 측정 |
| 사용자당 주간 Paimy 인터랙션 수 | 활성 사용도 |
| 이메일→태스크 전환율 | Gmail 연동 효과 측정 |
| 캘린더 기반 일정 생성 건수 | Calendar 연동 활용도 |

## 3. 타겟 사용자 (Target Users)

### 3.1 1차 사용자

- **PM / PO**
    - 다수의 프로젝트와 액션 아이템을 관리
    - 상태 업데이트 및 리마인드에 높은 피로도
    - 이메일과 미팅에서 발생하는 액션 아이템 추적 필요

### 3.2 2차 사용자

- **프로젝트 참여자 (개발, 기획, 운영)**
    - 슬랙 중심으로 업무 커뮤니케이션
    - 노션 직접 업데이트에는 소극적
    - 미팅 일정 확인 및 조율 필요

## 4. 핵심 사용자 가치 (User Value Proposition)

| 사용자 | 제공 가치 |

## 5. 핵심 기능 범위 (In Scope)

### 5.1 슬랙 기반 반응형 관리 (Pull)

- 슬랙에서 자연어로:
    - 개인/팀 태스크 조회
    - 조건 기반 태스크 검색
    - 태스크 상태 및 속성 변경
- 대화 맥락을 이해한 태스크 식별

### 5.2 선제적 프로젝트 운영 (Push)

- 일일 모닝 브리핑 (태스크 + 미팅 일정 + 중요 메일 통합)
- 주간 프로젝트 요약 리포트
- 마감 임박 및 지연 태스크 리마인드
- 상태 업데이트 유도 메시지 제공

### 5.3 협업 지원

- 태스크 기반 관계자 식별
- 슬랙 멤버 태깅을 통한 커뮤니케이션 연결

### 5.4 Google Calendar 연동

- **일정 조회**: 개인/팀원의 일정 확인, 특정 기간 미팅 검색
- **일정 생성**: 자연어로 미팅 생성 및 참석자 초대
- **일정 수정/삭제**: 기존 일정 변경 및 취소
- **가용 시간 확인**: 미팅 시간 조율을 위한 빈 시간대 탐색
- **태스크-일정 연계**: 태스크 관련 미팅 자동 연결

### 5.5 Gmail 연동

- **메일 요약**: 읽지 않은 중요 메일 요약 제공
- **메일 검색**: 조건 기반 메일 검색 (발신자, 기간, 키워드)
- **액션 아이템 추출**: 메일 본문에서 할 일 자동 식별
- **메일→태스크 전환**: 추출된 액션 아이템을 노션 DB에 자동 생성

### 5.6 크로스 플랫폼 자동화

- **통합 모닝 브리핑**: 오늘의 미팅 + 마감 태스크 + 중요 메일 통합 요약
- **메일→노션 태스크화**: Gmail 액션 아이템 감지 → 승인 후 노션 DB 등록
- **태스크→미팅 생성**: 태스크 담당자 기반 캘린더 미팅 자동 생성
- **미팅 후 액션 아이템 기록**: 미팅 종료 후 액션 아이템 입력 유도

## 6. 비목표 (Out of Scope)

- ~~노션 DB 연동~~ → 필수 범위에 포함

- 프로젝트 성과 예측 및 고급 분석
- Gmail 자동 답장 작성
- 캘린더 일정 자동 최적화/재배치

## 7. 사용자 경험 원칙 (UX Principles)

1. **슬랙 퍼스트(Slack-first)**
    
    → 노션, 캘린더, Gmail을 열지 않아도 대부분의 관리가 가능해야 함
    
2. **Zero-UI**
    
    → 버튼보다 대화, 설정 대신 자연어
    
3. **즉각적 피드백**
    
    → 모든 변경은 결과를 명확히 알려야 함
    
4. **사용자 확인 우선**
    
    → 자동 생성/변경 전 반드시 사용자 승인 절차 포함

## 8. 릴리즈 전략 (Phased Rollout)

| Phase | 범위 | 예상 기간 |
|-------|------|----------|
| **Phase 1** | 슬랙-노션 코어: 태스크 조회/상태 변경, 일일·주간 브리핑, 마감 리마인드 | 4주 |
| **Phase 2** | Google Calendar 연동: 일정 조회/생성/수정, 통합 브리핑 | 3주 |
| **Phase 3** | Gmail 연동: 메일 요약/검색, 액션 아이템 추출, 메일→태스크 자동화 | 4주 |
| **Phase 4** | 크로스 플랫폼 인텔리전스: 컨텍스트 인식 강화, 협업 추천 정교화, 개인화 알림 설정 | 3주 |

## 9. 리스크 및 고려사항

### 9.1 기존 리스크

- 자연어 해석 오류에 따른 잘못된 상태 변경
- 슬랙–노션 ID 매핑 정확성
- 권한 관리 및 변경 이력 추적 필요성

### 9.2 GWS 연동 관련 추가 리스크

| 리스크 | 대응 방안 |
|--------|----------|
| **OAuth 인증 복잡성** | 사용자별 GWS 인증 플로우 간소화, 토큰 갱신 자동화 |
| **메일 프라이버시 우려** | 메일 본문 접근 범위 명확화, 데이터 보존 정책 수립, 사용자 동의 절차 |
| **액션 아이템 추출 오류** | 자동 태스크 생성 전 슬랙에서 사용자 확인 단계 필수화 |
| **일정 충돌/중복 생성** | 미팅 생성 전 기존 일정 확인 로직, 중복 방지 알림 |
| **API 호출 제한** | Google API 쿼터 모니터링, 캐싱 전략 수립 |
